## 04:48 — claude

**Change:** Camera "Test connection" now surfaces the REAL ffprobe failure reason instead of an opaque `ffprobe exit 1`. `ffprobeStream` was running `ffprobe -v quiet` and discarding stderr, so any live-stream (SETUP/PLAY) rejection collapsed to `UNSTABLE_STREAM` / `ffprobe exit 1` with no cause — undiagnosable in the field. Switched to `-v error`, captured stderr, and surface the last stderr line (redacted) as `errorMessage`.

**Files:** `backend/src/modules/health/health.checkers.ts`, `backend/src/modules/health/health.checkers.test.ts`

**Reason:** User report — "unable to add a camera, Test connection fails with `Probe failed / DESCRIBE: ok · Video: ffprobe exit 1`; find and fix it" + "make sure it is production ready / works in production". Root cause is diagnostic opacity, NOT the URL/credential handling.

**Root cause (evidence-based, systematic-debugging):**
- DESCRIBE succeeds (`DESCRIBE: ok`) → host reachable, RTSP port open, credentials accepted. Failure is strictly in the ffprobe video stage (SETUP/PLAY / stream read).
- Disproved my first hypothesis (percent-encoding `%40` breaking auth) against the real `ffprobe 8.0.1` binary via a TCP RTSP mock: ffmpeg correctly decodes `%40`→`@` in userinfo AND preserves the vendor path (`/user=…_password=…_channel=…`) byte-for-byte. `injectRtspCredentials` is correct — did NOT touch it.
- The DEFECT: `ffprobeStream` used `-v quiet` and reported only `ffprobe exit ${code}`, throwing away the one line that says WHY (`Server returned 401 Unauthorized`, `method SETUP failed: 461 Unsupported Transport`, `Invalid data found when processing input`, `Connection timed out`, …).

**Fix details:**
- `-v quiet` → `-v error`: stdout stays clean JSON (`-of json`) on success; stderr carries the concrete cause on failure.
- Capture `proc.stderr`; new `ffprobeFailureReason(stderr, code)` helper takes the last non-empty stderr line, runs it through `redactRtsp()` (scrubs userinfo + Dahua/Hik `user=`/`password=` path tokens + Authorization/Bearer/Basic), strips the leading redacted `rtsp://…: ` URL prefix so only the human reason remains, caps at 300 chars, falls back to `ffprobe exit <code>` if stderr is empty.
- `errorCode` UNCHANGED (`UNSTABLE_STREAM`) → health state machine / scheduler behavior untouched; this is a message-enrichment change only.
- Zero frontend change needed: `ConfigureCameraModal` already renders `probe.video.errorMessage ?? errorCode`, so the real reason now shows on the next Test connection.

**Security:** stderr echoes the full credentialed URL — redaction happens BEFORE the string is returned/surfaced/asserted. Test proves no credential shape (userinfo, `password=`, username, `@`) leaks into the message. No plaintext creds logged; no real creds/IPs used in tests (RFC 5737 `198.51.100.7`, fabricated password).

**Tests added/updated:** yes — `health.checkers.test.ts`: added `stderr` EventEmitter to `fakeProc`; 3 new cases: (1) extracts + redacts real stderr reason on non-zero exit, (2) generic-exit fallback when stderr empty, (3) success path parses stdout JSON (guards the `-v error` switch).

**Root cause (evidence-based):**
- DESCRIBE consistently green → reach + auth + creds all correct. Failure was strictly the ffprobe live-video stage.
- ffprobe/ffmpeg default RTSP transport is UDP; many cameras/NAT paths reject UDP SETUP → `461 Unsupported Transport` or silent `Connection timed out`. TCP-first with a bounded read timeout resolves the dominant class; UDP retry preserves the minority that only speak UDP.
- Even with transport fixed, a single-frame read can transiently fail on a correctly-wired camera (keyframe interval, transient bitrate) — blocking activation on that stranded operators with a green DESCRIBE and a red "Probe failed". Video is therefore advisory; continuous health monitoring owns the live verdict post-activation.
